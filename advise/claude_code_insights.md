## 问题分析

### 1. API 和网络不稳定
你经历了频繁的 API 超时和错误，阻碍了进度。考虑在开始复杂分析任务前检查网络连接和 API 可用性，或将工作分解为更小的检查点。

**示例：**
- 多个 524 错误中断了你的策略代码审查，最终阻止了调试会话的完成
- Binance API 调用因网络问题超时，需要多次脚本重写才能获得可用的数据下载

### 2. 表层修复先于根因分析
你有时从高层修复开始，但没有解决底层问题。让 Claude 先追踪完整调用栈或检查绑定层可以节省迭代周期。

**示例：**
- 持仓回调问题的初始 Python 层修复导致 TypeError，因为 C++ 绑定不接受 exchange 参数，需要更深入地研究 C++ 代码
- get_active_positions() 修复需要在更低层级使用 refresh_positions()——从"为什么返回空"开始而不是"修复这个行为"会更快

### 3. 数据格式假设不匹配
你遇到了意外的响应格式，破坏了初始实现。预先指定预期数据格式或让 Claude 先验证响应结构可以防止重写。

**示例：**
- 初始 XML 解析方法失败，因为 Binance 返回 HTML 而非预期格式，需要完全重写
- 需要多次脚本迭代才能正确处理 Binance K线数据——先请求格式验证步骤会更早发现这个问题
